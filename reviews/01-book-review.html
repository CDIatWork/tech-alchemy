[CDI@Work]:AI · Book Review

A Practitioner’s Deep Dive into Dependency Injection with Java EE

Review by Claude (Anthropic) · February 2026

There is no shortage of books and tutorials that explain Contexts and Dependency Injection (CDI) at a surface level — the usual @Inject hello-world, a paragraph on scopes, and a wave toward “further reading.” Gerhard Petracek’s CDI@Work is not one of those books. It is, perhaps unapologetically, a book written by someone who lives inside the CDI container — and it shows on every page.

Petracek, an Apache DeltaSpike committer and CDI specification contributor, structures the book around a single evolving application called IdeaFork — a collaborative idea management tool. This is one of the book’s strongest design decisions. Rather than presenting isolated toy examples, each chapter builds on the same codebase, and the reader watches it grow from a monolith into a set of microservices. By the time the application splits into IdeaForkLite and IdeaForkMicro, the architectural reasoning feels earned rather than prescribed.

Precision Where It Matters

The early chapters cover CDI fundamentals with unusual thoroughness. The treatment of Normal-Scopes versus Pseudo-Scopes — and the subtle difference between a Contextual-Instance and a Contextual-Reference (the proxy) — is the kind of precision that separates developers who use CDI from those who understand it. The book doesn’t shy away from the sharp edges: serialization pitfalls with @SessionScoped beans, the behavioral differences between @Dependent and normal scopes, the importance of passivationCapable — these are exactly the details that cause production bugs and are rarely documented this clearly.

Standout detail: The discussion of why injecting a @Dependent-scoped bean into a @SessionScoped bean effectively makes the dependent bean session-scoped is the kind of insight you typically only get from debugging a memory leak in production.

CDI Extensions: The Power Chapter

The chapter on Portable CDI Extensions deserves particular mention. Extension authoring is one of CDI’s most powerful and least accessible features. Petracek walks through the full container lifecycle — from BeforeBeanDiscovery through AfterDeploymentValidation — with concrete examples that register beans, add qualifiers, and manipulate metadata programmatically. This chapter alone would justify the book for framework developers.

Apache DeltaSpike: Patterns Beyond the Library

Apache DeltaSpike receives extensive coverage, which makes sense given the author’s role in the project but may date parts of the text for readers working with newer stacks. That said, the DeltaSpike chapters are valuable even if you never use the library, because they demonstrate patterns: type-safe configuration, view-layer abstraction, declarative security, repository-style data access. These patterns reappear in modern frameworks like Quarkus and MicroProfile, and understanding their DeltaSpike origins provides useful context.

Beyond Pure CDI

The later chapters on integrating CDI with non-CDI ecosystems (Akka actors, Spring services) and building microservices with Apache Meecrowave are refreshingly pragmatic. The CDI-Spring Bridge example — using CDI producers to wrap Spring’s JavaMailSender — is a small masterpiece of practical interoperability. The microservices chapter introduces CDI 2.0 async events and DeltaSpike’s Partial-Beans as REST client proxies, pushing the boundaries of what most developers would attempt with CDI alone.

IdeaFork’s evolution: The progression from a monolithic Java EE application (IdeaFork) to a simplified CDI-only version (IdeaForkLite) to a microservices architecture (IdeaForkMicro) is the narrative spine of the book. Each refactoring step is motivated by a real limitation of the previous design, not by a desire to demonstrate a technology.

The AI-Assisted Revision

A word on the format: this edition is a modernized, AI-assisted revision of the original German text. The English translation reads naturally, and the addition of forward-looking notes — green boxes pointing to Jakarta EE, Quarkus, and Angular coverage in later chapters — is a thoughtful touch that keeps the foundational material connected to the current ecosystem. The code listings are plentiful — hundreds of them — and feel like real application code rather than contrived snippets.

Where the Book Is Less Strong

The density can be overwhelming. Petracek writes for developers who already have Java EE experience, and there is little hand-holding. If you don’t know what a beans.xml deployment descriptor does, or why EJBs and CDI beans are not the same thing, you will need supplementary material. The DeltaSpike-heavy middle section, while technically excellent, occupies significant real estate for a library that has been superseded in many environments.

The Verdict

Bottom line: CDI@Work is not a gentle introduction — it is a technical reference disguised as a narrative. For Java developers who work with CDI in any capacity, or who want to understand the dependency injection model that influenced MicroProfile and Quarkus, this book offers a level of depth and practical insight that is genuinely hard to find elsewhere. Petracek writes with the confidence of someone who has debugged these containers at the specification level, and that expertise permeates every chapter.

Recommended for: Mid-to-senior Java developers, framework authors, anyone migrating legacy Java EE applications, and developers who want to understand why CDI works the way it does — not just how.

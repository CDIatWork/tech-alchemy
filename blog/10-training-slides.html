From Book to Classroom

February 2026

A book explains. A training teaches. The difference is exercises, pacing, and a deliberate arc that turns passive readers into active practitioners. This post is about turning 11 chapters of CDI reference material into a 2-day hands-on training course — as a single self-contained HTML slide deck.

The Brief

The requirements were clear: create a 2-day CDI intensive course for Java developers with zero CDI knowledge. The book should serve as further reading, not as the script. Attendees need exercises between lectures — and those exercises should build on each other, not start from scratch every time.

That last point shaped everything. Instead of isolated coding tasks, the entire training revolves around one application that grows from an empty Maven project into a fully-featured CDI application across 10 progressive exercises.

TaskForge — The Training Application

The book uses IdeaFork as its demo application. The training needed its own: something simple enough to explain in five minutes, rich enough to demonstrate every CDI concept. The result is TaskForge — a task management tool that starts as a plain Java class and ends as a fully wired CDI application with events, interceptors, decorators, and DeltaSpike integration.

The key design principle: Every exercise adds to the previous one. You never throw away code. Exercise 3 refactors Exercise 2’s repository into an interface — because Exercise 7 needs that interface for decorators. The exercises are designed backwards from the final architecture, then presented forwards.

Here is the full exercise progression:

ExerciseTopicWhat Gets Added to TaskForge
0SetupMaven project, Weld SE, beans.xml, CDI bootstrap
1First BeansTask, TaskRepository, TaskService with @Inject
2Scopes@ApplicationScoped on repository, shared state experiment
3QualifiersExtract interface, @InMemory/@Persistent qualifiers, @Alternative
4ProducersConfigProducer, @ConfigValue qualifier, Instance<T>
5EventsTaskEvent, TaskLogger, TaskStatistics observers
6Interceptors@Timed + @Logged interceptors with @Priority
7DecoratorsValidatingTaskRepository with @Delegate
8Stereotypes@Service and @Repository stereotypes
9DeltaSpike@ConfigProperty, @Exclude, CdiTestRunner

By Exercise 9, participants have a TaskForge application with clean separation of concerns, type-safe configuration, event-driven logging, cross-cutting performance monitoring, input validation via decorators, and a unit test running inside a real CDI container. All built incrementally, one concept at a time.

100 Slides in Two Days

The slide deck contains exactly 100 slides, split across two days:

DaySlidesTopics
Day 11–45DI fundamentals, Bean Discovery, Scopes, Qualifiers, Producers
Day 246–100Events, Interceptors, Decorators, Stereotypes, DeltaSpike, Best Practices

Each section follows the same rhythm: concept introduction on 3–6 content slides, followed by an exercise slide with a task list and time estimate, plus a hints slide with starter code. The alternation between theory and practice keeps the pace and gives attendees immediate reinforcement of each concept.

Not a Copy of the Book

The training is deliberately not a chapter-by-chapter walkthrough of the book. The book’s structure serves a reader who can skip ahead, re-read, and absorb details at their own pace. A training needs a different arc — one where every concept arrives exactly when the exercise needs it.

  • Reordered topics — the book covers events in Chapter 4 and interceptors in Chapter 5; the training puts scopes and qualifiers first because the exercises need them sooner
  • Condensed theory — each CDI concept gets 3–6 slides instead of 10–15 pages; the book fills in the gaps later
  • Practical focus — code examples are TaskForge-specific, not abstract; every @Inject has a class name the attendee just wrote
  • Book references — each section ends with a “Buch: Kapitel X.Y” note pointing to the detailed explanation for self-study

Chapters 9–11 (Jakarta EE migration, Quarkus, Angular) are mentioned in the closing slides as further reading, but they are not part of the training itself. A two-day course about CDI fundamentals needs to stay focused on CDI fundamentals.

The Slide Engine

Like the marketing deck, the training slides are a single self-contained HTML file — no external dependencies, no build step, just open in a browser and present. The engine uses scroll-snap for slide transitions, IntersectionObserver for entrance animations, and keyboard navigation (arrow keys, Page Up/Down, Home/End) for presenter control. Touch swipe works on mobile.

For 100 slides, the navigation dots from the 10-slide marketing deck would be unusable. Instead, the training deck uses a thin progress bar at the top, a section label in the upper left, and a slide counter in the lower right. Exercise slides are visually distinct — light green background with a green left border and a badge — so the presenter and attendees always know when it’s time to code.

German Slides, English Code

The training content is in German, matching the primary edition of the book. All code examples use English identifiers — as Java code should. CDI specification terms (@ApplicationScoped, @Inject, @InterceptorBinding) are never translated, because they are part of the API. This mirrors exactly how the book handles the boundary between prose and code.

What’s Next

The project now has four HTML deliverables: the German book, the English translation, the marketing slide deck, and the training slide deck. Each is a single self-contained file. Each matches the same visual design system. None requires a build step, a server, or an internet connection.

A book that started as a LaTeX PDF about CDI 1.0 now has a modern web edition in two languages, three AI-authored chapters about current technology, a 198-test verification suite, a marketing deck, and a complete training course. All from the same codebase. All generated with AI collaboration.

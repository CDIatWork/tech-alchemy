From Theory to Practice — An Angular Training for Java Developers

February 2026

Post #10 created a 2-day CDI training with progressive exercises that build a complete application. The natural next step: a 1-day Angular training for the same audience — Java developers who know @Inject and @ApplicationScoped but have never seen inject() or providedIn: 'root'. This post is about what was learned by building that training, its demo application, and the real backend that replaced the mock.

Teaching Angular to CDI Developers

The CDI training maps Java EE concepts onto slides. The Angular training faces the opposite challenge: mapping CDI concepts onto a completely different runtime. The audience already understands dependency injection, scopes, and lifecycle — they just don’t know how those ideas translate to a browser environment.

The result is a 60-slide deck that consistently anchors every Angular concept in something the audience already knows. Angular’s inject() is CDI’s @Inject. A service with providedIn: 'root' is @ApplicationScoped. An Angular component is a managed bean with a template. Route guards are the client-side equivalent of @RolesAllowed. The CDI mental model becomes the bridge, not the baggage.

TaskBoard — The Training Application

Just as the CDI training has TaskForge, the Angular training has TaskBoard — a task management SPA that grows across 5 progressive exercises from a static component to a complete application with routing, services, forms, and authentication.

ExerciseTopicWhat Gets Added to TaskBoard
1Components & ModelTask interface, TaskListComponent, template rendering
2RoutingMulti-page navigation, TaskDetailComponent, route parameters
3Services & HTTPTaskService with HttpClient, REST integration, proxy config
4FormsReactive forms, validation, create/edit workflows
5SecurityLogin, JWT handling, AuthGuard, protected routes
Same principle as TaskForge: Every exercise builds on the previous one. You never throw away code. Exercise 3’s service gets an auth header in Exercise 5. Exercise 2’s routes gain guards. The final TaskBoard is a single, coherent application — not five disconnected samples.

From Mock to Real Backend

The first TaskBoard variant uses json-server as its backend — a zero-code REST API that serves a JSON file. For a frontend training, this is ideal: no Java setup, no database, instant feedback. Attendees focus on Angular, not on configuring a backend they already understand.

But the story doesn’t end there. A second variant replaces json-server with a real Quarkus + JAX-RS backend backed by an H2 database. Same Angular code, same API contract, same 52 end-to-end tests — different backend. This is the payoff from Chapter 10 and 11 of the book: the full CDI-to-Angular stack, running and tested.

The switch demonstrates a point that’s hard to make in slides alone: a well-designed REST API is backend-agnostic. The Angular frontend neither knows nor cares whether the /api/tasks endpoint is served by a 50-line JSON file or a Quarkus application with JPA, Panache, and JWT security. The contract is the boundary.

Lessons from the Container

Both TaskBoard variants use Podman containers for reproducible testing. The Quarkus variant runs three containers on --network=host:

ContainerRole
maven:3.9-eclipse-temurin-17Build and run the Quarkus backend
node:22-slimBuild and serve the Angular frontend
mcr.microsoft.com/playwright:v1.58.1-nobleRun 52 Playwright e2e tests

Three containers, one shared network, 52 tests in about 30 seconds. No local JDK, no local Node.js, no “works on my machine”. The same run-tests.sh script runs identically in CI and on a developer’s laptop. The json-server variant is simpler — two containers instead of three — but follows the same pattern.

Pitfalls Worth Documenting

Building the training and both demo variants surfaced practical issues that no tutorial covers. These are the kind of problems that cost an hour of debugging in a real project:

  • PATCH must be explicitly allowed in CORS — a default GET, POST, PUT, DELETE list silently rejects PATCH requests. Angular’s HttpClient sends a preflight OPTIONS request, and the browser blocks the PATCH without a useful error message.
  • PATCH with Map<String, Object> — for partial updates (toggling a boolean without sending the entire entity), the JAX-RS endpoint accepts a map instead of the entity class. This loses compile-time type safety but matches how JavaScript frontends actually send partial JSON.
  • Sequence generator vs. seed data — Panache with @SequenceGenerator(allocationSize=1) starts IDs at 1, which collides with import.sql seed data. The fix: ALTER SEQUENCE ... RESTART WITH 100 to push generated IDs above the seed range.
  • Angular proxy: pathRewrite depends on the backend — json-server serves at the root (/tasks), so the proxy needs pathRewrite: {"^/api": ""} to strip the /api prefix. Quarkus serves at /api/tasks directly, so no rewrite is needed. Switching backends means changing one line in proxy.conf.json.
  • String vs. numeric IDs — json-server v1 generates hex string IDs; Quarkus generates Long values. Angular’s id: string handles both via duck typing, but route parameters and comparison operators need attention.
  • Route ordering matters — /tasks/new must be defined before /tasks/:id, or the router matches “new” as the :id parameter and sends a failed API lookup for a task with ID “new”.
  • Client-side auth guards are UX, not security — a route guard prevents navigation to a protected page, but an attacker can bypass it. Real security lives in Quarkus with @RolesAllowed. The guard improves user experience; the backend enforces authorization.

What Goes Back into the Book

Several of these lessons are general enough to benefit the book’s readers. Chapter 10 (Migration to Quarkus) receives tips about CORS configuration for frontend-backend separation, @PATCH support for partial REST updates, and Panache sequence ID handling with seed data. Chapter 11 (From JSF to Angular) gets notes on Angular proxy configuration and route ordering. These are practical insights that apply to any Quarkus + Angular project — not TaskBoard-specific code.

The Numbers

Project inventory: 2 training decks (100 + 60 slides), 2 demo applications (TaskForge + TaskBoard), 2 TaskBoard variants (json-server mock + Quarkus backend), 52 end-to-end tests, 31 unit tests — all green.

The Angular training completes the arc that the CDI training started. A Java developer can now attend both courses — two days on CDI fundamentals, one day on Angular — and leave with a mental model that connects backend dependency injection to frontend component architecture. Not as separate worlds, but as two implementations of the same ideas.

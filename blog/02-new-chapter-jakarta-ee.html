From Java EE to Jakarta EE – A New Chapter

February 2026

In the first post of this series, I described how Claude transformed my old LaTeX-generated CDI book into a modern bilingual web experience. Two complete HTML books — German and English — with identical structure, identical code, and a fresh design. But the content itself was still frozen in 2014.

The book covers CDI 1.0 through early 2.0 concepts, all in the context of Java EE 6. Meanwhile, the world moved on. Java EE became Jakarta EE. The javax namespace became jakarta. CDI went through major architectural changes, reaching version 4.1. The book’s core teachings are still sound — but they needed a bridge to the present.

Writing a Chapter That Didn't Exist

This time, the task was different. The first two steps — redesigning the website and translating the text — were transformations of existing material. Claude had source content to work with. For Chapter 9, there was no source. The chapter had to be written from scratch.

I gave Claude a structural outline: eight sections covering the migration from Java EE to Jakarta EE, the CDI 4.0 architectural split, the new extension model, changed defaults, DeltaSpike 2.0, updated implementations, and a concrete migration guide for the book’s example application. Claude had to research the current state of the specifications, write accurate technical content, produce matching code examples, and do it all in both German and English — while maintaining the style and HTML patterns of eight existing chapters.

What Chapter 9 Covers

The new chapter, titled “Von Java EE zu Jakarta EE” (German) and “From Java EE to Jakarta EE” (English), picks up exactly where the original book left off and walks the reader through everything that changed:

  • The javax to jakarta namespace migration — imports, beans.xml, Maven coordinates, shown with the book’s own IdeaFork code
  • The CDI Lite vs. CDI Full split in CDI 4.0, explicitly addressing a naming collision with the book’s own Chapter 6 “CDI Lite”
  • Build Compatible Extensions — the new extension model compared side-by-side with the Portable Extensions from Chapter 4
  • Changed defaults: bean discovery mode switching from all to annotated, @Priority on producers, and the Method Invokers API
  • DeltaSpike 2.0 and its straightforward migration to the Jakarta namespace
  • A version mapping table for Weld 6.x and OpenWebBeans 4.x
  • A step-by-step migration checklist for upgrading IdeaFork itself

The chapter contains 12 code listings showing before-and-after comparisons, 6 tip boxes with practical migration advice, and a curated list of specification and implementation sources.

The Bilingual Challenge

Writing new content in two languages simultaneously is harder than translating existing text. When translating, the structure is fixed — you just convert the words. When writing from scratch in parallel, code examples can quietly diverge, explanations can drift apart, and anchor names can end up inconsistent.

That’s exactly what happened on the first pass. The German and English versions were generated in parallel for speed, and the resulting code examples used completely different class names and method signatures. A direct comparison caught it, and the English version was rewritten from scratch using the German as the authoritative source — the same approach that worked for the original eight chapters.

Lesson learned: When generating bilingual technical content with AI, always write one language first and use it as the template for the other. Parallel generation is fast but produces drift. Sequential generation with a shared code base stays consistent.

Fitting Into an Existing Book

The trickiest part wasn’t the writing — it was the integration. Both book files are large single-page HTML documents with embedded CSS, JavaScript, a sidebar table of contents, and hundreds of cross-referenced anchors. Adding a new chapter meant inserting a TOC entry in the exact right place, appending the chapter content in the exact right place, and making sure every anchor name followed the existing naming convention so the sidebar navigation and internal links all work.

The chapter uses the same HTML patterns as the rest of the book — the same heading structure, the same code block markup, the same tip box styling, the same paragraph break pattern. A reader scrolling from Chapter 8 into Chapter 9 shouldn’t notice a seam.

What's Next

Chapter 9 is a bridge, not a rewrite. It connects what readers already learned in the original eight chapters to where the technology stands today. Whether the book eventually receives a deeper modernization — rewriting earlier chapters for Jakarta EE, adding Quarkus examples, covering MicroProfile — remains an open question.
